|---|---|---|

| `POST /run/room` | — | Process chest/trap |
| `POST /run/battle-result` | `{won, damageTaken, damageDealt, skillUses, actualHpLost, potionsUsed}` | After fight |

| `POST /character/skills` | `{skills: string[]}` | Save equipped skills (max 2) |

**Prisma Character fields:**
`level, energy, endurance, strength, agility, luck, gold, trophies, crystals, potionCharges, totalDamageReceived, totalDamageDealt, totalSkillUses, strengthAtLevelUp, enduranceAtLevelUp, equippedSkills([]), currentRun(Json?)`

**`currentRun` shape:** `{ rooms, index, hp, potions, puzzleId? }`

### Frontend (src/)

**`Battle.tsx`** — PixiJS v8 fullscreen combat.
Props: `{initialHp, maxHp, isBoss?, level?, equippedSkills?, potionCharges?, strength?, onBattleEnd}`

**⚠️ Спрайты игрока — УСТАРЕВШИЙ раздел, см. "Art & Animation Pipeline" ниже.**
Старые заглушки (Walk.png 8 кадров 128×128 и т.д.) заменены полным набором
анимаций героя и первого врага, ещё не подключённым к коду. Секция ниже
оставлена как есть до момента интеграции нового набора в Battle.tsx.

**Координаты в Battle.tsx:**
- FLOOR_Y = height - 200
- player.y = FLOOR_Y (anchor = 0.5, 1 — ногами на полу)
- enemy.y = FLOOR_Y - ENEMY_H + 40
- WORLD_WIDTH = 3000, камера следует за игроком

**Анимации игрока (реализовано в коде на момент последней проверки):**
- idle: idleFrames, speed 0.15, когда стоит
- walk: walkFrames, speed 0.3, когда движется
- attack: attackFrames, speed 0.4, один раз при нажатии ⚔, потом возврат к idle/walk
- Флип: scale.x отрицательный = смотрит влево, положительный = вправо

**Параллакс в ticker:**
- bgSky.tilePosition.x = -cameraX * 0.1
- bgRuins.tilePosition.x = -cameraX * 0.3
- platform.tilePosition.x = -cameraX

**`App.tsx`** — Main state.
PlayerData: `{id, firstName, level, gold, trophies, strength, endurance, agility, equippedSkills, potionCharges}`
- Navigation: 5-tab bottom nav (Персонаж / Магазин / Исследовать / Снаряжение / Друзья)
- Run flow: Start → showRoomIntro(2s) → enterCurrentRoomDirect → auto-next → results screen

## Art & Animation Pipeline (НОВОЕ — готово, ждёт интеграции в код)

### Пайплайн (проверенный, воспроизводимый)
```
1. ДИЗАЙН: Nano Banana Pro (Google Flow) → референс-арт персонажа,
   строго вид сбоку ("side view profile, facing right/left"), плашка сзади.
2. АНИМАЦИЯ: Veo (через Flow) → видео 4с/24fps по 4-блочному скелету промпта:
   a) вид сбоку + статичная камера + "never turns toward camera"
   b) кадрирование (wide framing, full body, never touches frame borders)
   c) ДЕЙСТВИЕ — единственный блок, который меняется между анимациями;
      описывать ГЕОМЕТРИЮ движения (что сгибается, куда смещается),
      НЕ эмоцию ("flinches" → модель рисует морду, а не тело)
   d) "Only the character on a flat gray background. No ground, no shadow,
      no dust, no effects" — жёсткий запрет вспышек/пыли/колец на ударе
3. НАРЕЗКА: ffmpeg → кадры → Python (PIL + scipy.ndimage) →
   удаление фона, поиск петли цикла, сборка спрайт-листа + PNG по кадрам.
```
Инструменты нарезки — одноразовые Python-скрипты в чате (не в репо), логика
описана ниже на случай повтора процесса для новых персонажей/анимаций.

### Известные грабли пайплайна (не наступать повторно)
- **Veo даёт секунды статики в начале** видео перед началом действия —
  всегда резать по скачку "движения между кадрами" (diff масок), а не брать
  видео от 1 кадра.
- **Слова "hit/strike/blow/dies/fatal"** триггерят фильтр Flow как опасный
  контент. Обход: переформулировать как физическое состояние без источника
  ("is exhausted and loses his strength" вместо "takes a fatal blow").
- **Модель дорисовывает эффекты, если в тексте есть "причина"** (получил
  удар → чем-то ударили → рисует снаряд/вспышку). Решение — убрать саму
  причину из промпта ("nothing touches it", "there is no ground beneath
  him"), а не только запрещать эффект.
- **Композиция против водяного знака**: знак Flow всегда в правом нижнем
  углу кадра. Если персонаж длинный (четвероногий, лежащий) и дотягивается
  до знака — они сливаются в один объект при вырезании фона. Фикс:
  в промпте явно просить "small in the frame, positioned in the upper
  left, large empty space in the bottom right corner" — тогда знак остаётся
  на пустом фоне и вырезается сам как отдельный компонент.
- **Заливка дыр (binary_fill_holes) не должна быть безусловной**: если
  заливать ВСЕ дыры в силуэте, крупные проймы (между рукой и телом, между
  ног) закрашиваются фоновым серым и остаются видимыми пятнами в игре на
  цветном фоне. Правильно — заливать только мелкие дыры (< ~400 px,
  артефакты контура), крупные оставлять прозрачными.
- **Выравнивание кадров по ступням годится ТОЛЬКО для кадров, сгенерированных
  по отдельности** (напр. спрайт-лист Nano Banana, где персонаж каждый раз
  в новом месте клетки). Для кадров, вытащенных из ОДНОГО видео со статичной
  камерой — кадры уже согласованы между собой; выравнивание по ступням
  каждого кадра отдельно портит анимацию (перса начинает трясти), т.к. якорь
  «ступни» гуляет из-за плаща/лапы/меча внизу силуэта. Правильно: резать все
  кадры одной анимации ОДНИМ неподвижным окном.
- **Между анимациями одного персонажа** (idle/attack/hurt и т.п.) якорь
  нельзя брать по ступням наивно — если в одной из анимаций конечность
  выносится далеко (бросок лапой, выпад мечом), медиана ступней уезжает и
  персонаж прыгает при смене состояния в движке. Правильно — совмещать
  анимации между собой по IoU-наложению силуэтов первого кадра с базовой
  (idle), а не по ступням.
- **Фон не всегда получается чисто зелёным** — иногда Veo даёт зеленовато-
  серый (менее насыщенный) или вообще серый фон (если в промпте забыто
  слово "green"). Простой порог `g > r+15 and g > b+15` в таком случае не
  добирает до alpha=0 в углах (остаётся дымка ~40-50 альфы). Надёжнее:
  сэмплировать РЕАЛЬНЫЙ цвет фона из угла КАЖДОГО кадра и вырезать по
  Евклидову расстоянию до него, а не по фиксированной "зелёности".
- **Разные анимации одного перса, снятые разными роликами Veo, не обязаны
  давать одинаковый bbox** — если в одной анимации конечность вылетает
  далеко за пределы позы из другой (напр. замах когтями шире, чем поза
  Idle), нельзя силой сажать оба в одинаковый квадратный холст: тело либо
  сожмётся мельче на кадрах с широким жестом, либо обрежется. Решение —
  единый МАСШТАБ пикселей (тот же коэффициент src→output, что и у Idle),
  но собственный (более широкий) холст под конкретную анимацию — как уже
  делали для Dash Strike героя (240×128 вместо 128×128).
- **Veo иногда разворачивает персонажа на 180° по горизонтали** несмотря на
  прямой запрет "keep facing direction", причём непредсказуемо — то в
  середине ролика на "активном" глаголе (rears back, атака), то сразу с
  первого кадра. Разворот ловится численно: найти на исходном референсе
  устойчивый асимметричный признак (напр. светящийся глаз) и сравнить его
  положение (% от края bbox) между роликами. Если только силуэт развернут,
  а само движение (напр. замах) логично относительно новой стороны —
  чинится зеркалом (`FLIP_LEFT_RIGHT`) при нарезке. Если же движение
  внутри ролика само по себе идёт "не в ту сторону от головы" (лапа бьёт
  НАЗАД от направления взгляда) — это не лечится зеркалом (оно перевернёт
  и голову, и лапу вместе, сохранив их неверное соотношение) — нужна
  перегенерация с явным "arm moves in the direction the head is facing".
  **Важно:** прежде чем чинить зеркалом — спросить у Андрея, видит ли ОН
  разворот на своём экране. Одного численного анализа недостаточно для
  решения переделывать готовый ролик.
- **"front-facing" и "profile/side-view" в одном промпте — противоречие.**
  Модель может послушать "front-facing" и развернуть персонажа лицом в
  камеру. Для строгого вида сбоку — только "strict side-view profile",
  без слова "front-facing" рядом.
- **Зум-наезд в первые ~5-10% ролика** — отдельный (от разворота) артефакт:
  Veo иногда "наезжает" от уменьшенного кадра до полного масштаба в первые
  8-10 кадров, даже при явном "no zoom" и стартовом изображении-референсе.
  Лечится не перегенерацией, а обрезкой этих кадров при нарезке — ищется
  кадр, где bbox стабилизируется (перестаёт расти), и нарезка начинается
  с него.
- **Референс-изображение "в упор" (объект во весь кадр) провоцирует Veo
  рисовать ещё крупнее и обрезать края.** Перед генерацией видео стоит
  всегда делать "отдалённый" референс-кадр: объект/персонаж на ~35-60%
  высоты холста 1920×1080, с полями со всех сторон, тем же зелёным фоном.
  Правильную долю (35% / 48% / 58% / 62%...) иногда приходится подбирать
  по обратной связи ("сильно отдалил" / "не влезает") — единого числа нет,
  зависит от пропорций объекта.
- **Копирайт-фильтр Google может сработать на силуэт, похожий на чужую IP**
  ("не могу создать видео из-за интересов сторонних поставщиков контента").
  Так сработало на дизайн NPC (сутулая фигура в капюшоне с 2 светящимися
  точками-глазами → похоже на Джаву из Star Wars). Обход — не переделывать
  дизайн полностью, а увести силуэт подальше (рост, форма капюшона,
  асимметричные/другой формы глаза) ИЛИ просто попробовать другой,
  более короткий промпт для видео с тем же PNG (иногда помогает).
- **Зелёный VFX-эффект на зелёном хромакее — плохая идея** (Heal-аура).
  Для таких случаев — чёрный фон + альфа = яркость пикселя (аддитивный
  ключ), а не хромакей по цвету.
- **Обычный экшн-глагол типа "rears back"** может сам по себе читаться как
  "поворачивается" — безопаснее описывать голой геометрией конечности
  ("pulls arm back behind itself"), а не позой всего тела.
- **Тень персонажа отличается от фона всего на ~10-40 по яркости** (почти
  тот же серый), а капюшон/тёмная одежда — на ~70-80. Фильтр "убрать тень
  по разнице с фоном" должен захватывать диапазон (-42..-6), а не любой
  тёмный объект, и применяться ТОЛЬКО к нижней трети кадра — иначе съедает
  тёмные детали костюма (было: капюшон героя лысел в анимации урона).

## Stat & Leveling System

### Growth (IMPLEMENTED)
- **Endurance**: normalized dmg received / (1+0.12×(lvl-1)). +1 per 80 (until End=30), then +1 per 250
- **Strength**: normalized dmg dealt / (1+0.18×(lvl-1)). +1 per 150 (until Str=20), then +1 per 350
- **Agility**: from totalSkillUses. Threshold = 10 + agility×5
- Attack damage = 15 + Math.floor(strength / 2)

### Leveling
- **Method 1**: +3 Endurance OR +6 Strength since last level-up
- **Method 2**: boss kill → instant +1 level

## Maps System (ГОТОВО — 6/6 карт, ждут интеграции в код)

Все 6 карт пула собраны, проверены на проходимость и сданы одним пакетом
`RightPlace_maps.zip` (сетки .txt + слоты .json + README.md). Инструменты

**Критично при любой новой правке рельефа:** проверка обязана требовать
**2 тайла воздуха** над каждой точкой стояния (герой 128px = 2 тайла), а не 1.
Старые проверки этого не делали и давали ложноположительный "проходимо" —
несколько итераций карты F ловили именно эту ошибку.

### Особая механика: заваливаемый тайник (карта D)
Пустой ролл Смуглера → обломки `R` на `[1,21]` и `[2,22]` становятся `#`,
дно ямы `[2,23]` (где Смуглер) недостижимо. Два файла-состояния:
`map_D_tainik_OPEN.txt` / `map_D_tainik_SEALED.txt`, оба проверены (115/115, 116/116).

### Особая механика: Обелиски (карта F, заменяет Загадку)
10 точек-кандидатов на всех 4 ярусах карты. 1 обелиск в `idle` на старте забега
(рандомная точка) → удар → `burning` + старт события → 30 сек, чтобы ударить
ещё 3 (спавнятся одновременно, рандом из оставшихся 9) → провал = конец забега.
**Не решено:** максимальная дистанция между обелисками на F может не укладываться
в 30 сек при неудачном ролле — варианты (ограничить радиус спавна / увеличить
таймер / сбрасывать таймер за удар / оставить как хардкор) не выбраны.

### Не хватает по изначальному распределению
Смуглер должен быть на B и D (сделан только на D). Загадка — на B и F (на F
заменена обелисками, на B не сделана вообще). Если нужно 1:1 соответствие —
доделать слоты на B.

### Рендер карты
- Красивый рендер карты (как "mapA final") делает НЕ `render_lib.py` (он
  процедурный, даёт рябь), а модуль `src/mapRenderer.ts` — перенесён из
  редактора карт `_ref/map_editor.html` (Canvas2D).
- Ключ красоты: `drawSolid` берёт КУСОК текстуры masonry под каждый тайл
  (`drawImage` с обрезкой sx,sy,tp), а не ужимает весь PNG; обводка
  рисуется только по краям, смотрящим в воздух (air-проверки); `=`
  платформы берут нижние 45% текстуры.
- `renderMapToCanvas({grid, decor, tileSize}) -> HTMLCanvasElement`,
  который в `Explore.tsx` оборачивается в `Texture.from(canvas)` и
  показывается как фоновый Sprite. Коллизия отдельно из .txt сетки.
- Текстура masonry и декор-спрайты вшиты в `mapRenderer.ts` как base64
  (взяты из `_ref/map_editor.html`). Папка `_ref/` в `.gitignore`.

## Design Decisions
- Leveling OR not AND (Method 1: +3 End OR +6 Str+Agi combined)
- Puzzle = game-mechanics quiz
- Stat normalization by level factor
- Heal/potions don't affect Endurance growth (totalDamageTaken tracked separately)
- actualHpLost sent to server for correct HP carry between rooms
- Enemy base stats: normal 120HP/14dmg, boss 200HP/18dmg
- Wide arena (3000px) with camera follow
- Sprites face left by default → flipped with scale.x = -1 in code (⚠️ проверить
  на актуальность после интеграции нового набора — новый герой сгенерирован
  смотрящим ВПРАВО, новый зверь — ВЛЕВО; направление флипа в коде может
  потребовать пересмотра под новые ассеты)
